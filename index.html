01 / FORMS & LABELS

Forms & labels

Every form control must have a persistent, programmatically associated label. Placeholder text is not a label.

Level ALevel AAWCAG 1.3.1 · 3.3.2 · 1.3.5

Label association

Correct
<label for="email">Email</label>
<input id="email" type="email">
Never
<input type="email"
 placeholder="Email">

Disappears on input. Invisible to screen readers in error state.

Group related inputs

<fieldset>
  <legend>Shipping address</legend>
  <label for="street">Street</label>
  <input id="street" type="text" autocomplete="street-address">
  <label for="city">City</label>
  <input id="city" type="text" autocomplete="address-level2">
</fieldset>
  • Radio groups and checkbox groups must use <fieldset> + <legend>
  • The legend is announced before each individual label inside the group

Required fields

  • Add aria-required="true" on the input element
  • Show a visible indicator — asterisk (*) is standard. Explain once: "* required"
  • Never rely on color alone to show required state

Autocomplete attributes — WCAG 1.3.5

INFOPersonal data fields must carry autocomplete attributes so password managers and browsers can assist users with cognitive and motor disabilities.
FieldValue
Full nameautocomplete="name"
Emailautocomplete="email"
Phoneautocomplete="tel"
Street addressautocomplete="street-address"
Current passwordautocomplete="current-password"
New passwordautocomplete="new-password"

Correct input types

  • type="email" — mobile email keyboard + browser validation
  • type="tel" — mobile number pad
  • type="url" — URL keyboard layout
  • type="number" — numeric-only values
  • type="search" — adds clear button in some browsers

02 / ERROR HANDLING

Error handling

When something fails, users must be told what failed and how to fix it — in text, not color alone.

Level ALevel AAWCAG 3.3.1 · 3.3.2 · 3.3.3
ERRNever use color or border alone to indicate an error. Screen readers cannot perceive visual styling. Always pair visual cues with descriptive text.

Required markup pattern

<label for="email">Email address</label>
<input
  id="email"
  type="email"
  aria-required="true"
  aria-invalid="true"           ← set when validation fails
  aria-describedby="email-err"  ← links to error message
>
<span id="email-err" role="alert">Enter a valid email address</span>
Attribute reference
AttributePurpose
aria-invalid="true"Signals error state to screen readers
aria-describedby="id"Points to error message — read after the field label
role="alert"Announces error immediately without focus moving

Error message rules

  • Be specific — not "invalid input" but "Password must be at least 8 characters"
  • Place the message directly below the field that failed
  • On submit: move focus to the first failing field
  • Show an error summary at top of form with anchor links to each failed field
  • Never clear the user's entered value on error
  • Reset: aria-invalid="false" and remove error text once field passes
  • Provide format hints upfront where possible: "Use MM/DD/YYYY"

Validation timing

TimingUse whenCaveat
Real-timePassword strength, username availabilityDon't penalize before user finishes typing
On blurEmail format, date formatBest for formatted inputs
On submitFull reviewAlways run this even if real-time is active

Error summary — multi-field forms

<div role="alert" aria-labelledby="err-head">
  <h2 id="err-head">2 errors in this form</h2>
  <ul>
    <li><a href="#email">Email: enter a valid address</a></li>
    <li><a href="#dob">Date of birth: use MM/DD/YYYY</a></li>
  </ul>
</div>

03 / KEYBOARD

Keyboard navigation

Every task must be completable without a mouse. Unplug yours and try every flow.

Level AWCAG 2.1.1 · 2.1.2 · 2.4.3 · 2.4.7

Universal keys

KeyActionNotes
TabNext interactive elementButtons, inputs, links, widgets
Shift + TabPrevious elementReverse direction
EnterActivateSubmit, follow link, open dropdown
SpaceToggle / scrollCheckbox, radio, button — NOT links
Arrow keysNavigate inside widgetDropdown, radio group, tabs, slider
EscapeClose / cancelModal, dropdown, tooltip — return focus to trigger
Home / EndFirst / last itemLists, tab lists, grids
WARNEnter vs Space: Buttons activate with either. Links activate with Enter only — Space scrolls the page. Use <button> for actions, <a> for navigation.

Custom widget rules

  • Custom interactive elements need tabindex="0" to be keyboard-reachable
  • Grouped widgets (tabs, radio group) = one Tab stop. Arrow keys navigate inside. This is the roving tabindex pattern.
  • Never use tabindex above 0 — breaks natural tab order
  • Tab order must match visual reading order — never reorder via CSS only
  • Never write outline: none without a custom focus replacement

Roving tabindex pattern

// Active: tabindex="0"  |  Others: tabindex="-1"
tabBtn.addEventListener('keydown', (e) => {
  if (e.key === 'ArrowRight') {
    current.tabIndex = -1;
    next.tabIndex = 0;
    next.focus();
  }
});

Skip link — required on every page

<!-- First element in <body> -->
<a href="#main" class="skip-link">Skip to main content</a>
<main id="main">...</main>

.skip-link { position: absolute; transform: translateY(-100%); }
.skip-link:focus { transform: translateY(0); }

04 / TABLES

Tables

Tables are for data, never layout. Screen readers navigate cell-by-cell using header context.

Level AWCAG 1.3.1

Minimum required markup

<table>
  <caption>Q1 2025 sales by region</caption>  ← always required
  <thead>
    <tr>
      <th scope="col">Region</th>
      <th scope="col">January</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <th scope="row">North</th>  ← row header
      <td>$14,200</td>
    </tr>
  </tbody>
</table>

Element reference

Element / attrPurposeRequired?
<caption>Describes the table — read first by screen readersAlways
scope="col"Declares a column headerAlways
scope="row"Declares a row headerRow-labelled data
<thead><tbody><tfoot>Structural groupingAlways
headers attrLinks cells to multiple headers in merged tablesComplex tables

Interactive and selectable tables

  • Selectable rows: aria-selected="true/false" on each <tr>
  • Select all checkbox: aria-label="Select all rows"
  • Per-row checkboxes: aria-label="Select [row name]" — unique per row
  • Sortable columns: aria-sort="ascending" / "descending" / "none" on the <th>
  • Announce sort via aria-live: "Sorted by date, ascending"
Keyboard inside a data grid
KeyAction
TabMove between interactive cells / buttons
↑ ↓Navigate rows
← →Navigate columns
SpaceToggle row selection / checkbox
INFOWrap tables in <div style="overflow-x:auto">. The div scrolls, not the page. At 200% zoom, no horizontal scroll should appear.

05 / MODALS

Modals & dialogs

Modals must trap focus, respond to Escape, and return focus precisely. One of the most commonly broken patterns.

Level AWCAG 2.1.1 · 2.1.2 · 4.1.2

Required keyboard behaviour

  • On open: move focus to modal heading or first focusable element inside
  • Tab must cycle only within the modal — cannot escape to page behind it (focus trap)
  • Escape closes and returns focus to exactly the element that triggered it

Required ARIA markup

<div
  role="dialog"
  aria-modal="true"           ← marks background as inert for screen readers
  aria-labelledby="modal-title"
>
  <h2 id="modal-title" tabindex="-1">Confirm deletion</h2>
  <p>This action cannot be undone.</p>
  <button>Cancel</button>
  <button>Delete</button>
</div>
INFOPair aria-modal="true" with the inert attribute on the page background for full browser support.

Focus management pattern

let trigger; // always store ref before opening

function openModal(el) {
  trigger = el;
  modal.removeAttribute('hidden');
  modalHeading.focus();
}

function closeModal() {
  modal.setAttribute('hidden', '');
  trigger.focus(); // always return to exact trigger
}
ERRNever use role="dialog" on side panels that don't trap focus. Use role="region" with aria-label instead.
REF: A11Y Collective: Modal Accessibility · W3C APG Dialog Pattern
06 / FOCUS MANAGEMENT

Focus management

Where focus lands after interactions is as important as what receives it. WCAG 2.2 added new rules here.

Level AANew in WCAG 2.2WCAG 2.4.11 · 2.4.3 · 2.4.7

The three rules

When something opens
Move focus to it — heading, first input, or named container
When something closes
Return focus to the exact element that triggered it — stored by ref
Never move focus uninvited
Only move focus as a direct response to a user action. Never on toast, loading state, or background updates.

Focus routing reference

EventMove focus to
Modal opensModal heading (tabindex="-1")
Modal closesExact trigger element
Side panel opensPanel heading (tabindex="-1")
SPA route changePage <h1> or hidden focus target
Form errorFirst error message or failed field
Toast appearsDo NOT move focus — announce via aria-live only
Tab activatesDo NOT move into panel — user decides with Tab

WCAG 2.2 — focus must not be obscured (2.4.11)

WARNFocused elements must not be hidden behind sticky headers, banners, or chat widgets. Fix with scroll-padding-top.
html {
  scroll-padding-top: 80px; /* height of your sticky header */
}

:focus-visible {
  outline: 2px solid currentColor;
  outline-offset: 2px;
}

07 / ARIA

ARIA & live regions

ARIA supplements native HTML — it should be your last resort, not your first move.

Level ALevel AAWCAG 4.1.2 · 4.1.3
ERRFirst rule of ARIA: don't use it if native HTML does the job. <button> always beats <div role="button">. Native elements come with keyboard handling and screen reader support for free.

Common attributes

AttributeWhat it doesUse case
aria-labelNames element — no visible text neededIcon-only buttons
aria-labelledbyPoints to element that names this oneModals, regions
aria-describedbyAdditional description (read after label)Error messages, hints
aria-expandedOpen/closed stateDropdowns, accordions
aria-selectedSelection stateTabs, listbox
aria-pressedToggle stateBold/italic, filters
aria-hidden="true"Hides from screen readers entirelyDecorative icons, charts
aria-liveAnnounces dynamic changes without moving focusStatus, counts, toasts

Live region pattern

WARNNever announce by injecting a new element. Update text inside a pre-existing live region that lives in the DOM from page load.
<!-- In DOM from page load — empty until needed -->
<div id="sr-announce"
     role="status"
     aria-live="polite"
     aria-atomic="true"
     class="sr-only"></div>
function announce(msg) {
  const el = document.getElementById('sr-announce');
  el.textContent = ''; // clear first — forces re-read of same text
  setTimeout(() => { el.textContent = msg; }, 50);
}

Live region types

TypeUse when
aria-live="polite"Updates, messages, counts — 95% of cases. Waits for user to finish.
role="alert"Session expiry, connection lost, critical errors. Interrupts immediately.
role="status"Progress, "saved", connection state

sr-only utility class

.sr-only {
  position: absolute;
  width: 1px; height: 1px;
  padding: 0; margin: -1px;
  overflow: hidden;
  clip: rect(0,0,0,0);
  white-space: nowrap;
  border: 0;
}

08 / COLOR & CONTRAST

Color & contrast

Color must never be the only way information is conveyed. Contrast ratios are non-negotiable.

Level AAWCAG 1.4.1 · 1.4.3 · 1.4.11

Required contrast ratios

Content typeMinimum
Normal text (below 18pt / 14pt bold)4.5 : 1
Large text (18pt+ or 14pt bold+)3 : 1
UI components — input borders, focus rings3 : 1
Meaningful icons3 : 1
Focus indicators3 : 1 against adjacent colors

Color-only violations and fixes

ERRApplies to: error states, status badges, chart series, required fields, active/inactive states, link text in body copy.
ViolationFix
Error — red border onlyRed border + error icon + text message
Chart series — color onlyDifferent dashes + different marker shapes
Required field — asterisk onlyAsterisk + "* required" legend
Active nav — color onlyColor + bold + underline or border accent
Status badge — color onlyColor + text label: "Active", "Error"

Tooling

  • Colour Contrast Analyser — free desktop app, checks rendered UI (not design files)
  • Chrome DevTools — hover any element in inspector to see contrast ratio
  • axe DevTools — browser extension, catches failures in CI
  • WAVE — visual overlay, good for explaining failures to stakeholders
INFOAlways verify in the rendered browser. Browser rendering can differ from design tool values.
09 / TESTING

Testing checklist

Automated tools catch ~30–40% of issues. Manual testing is always required for the rest.

Manual protocol — run before every release

  • Unplug mouse — complete every primary task using keyboard only
  • Enable VoiceOver (Mac) or NVDA (Windows) — complete primary tasks
  • Zoom to 200% — check for horizontal scroll and overlapping content
  • Tab through all forms — verify every field has an announced label
  • Submit forms with empty/invalid inputs — verify errors are announced
  • Open and close every modal — check focus trap and focus return
  • Audit all icon-only buttons — confirm accessible names (not just "button")
  • Enable high contrast mode — check all states are distinguishable
  • Disable CSS — verify logical reading order matches visual order

Automated tools

ToolWhat it catchesWhere
axe DevToolsLabels, contrast, ARIA issuesBrowser ext + CI
LighthouseScore, contrast, labelsChrome DevTools
WAVEVisual overlay, structureBrowser ext / web
WARNAutomated tools cannot detect: missing focus management, incorrect keyboard patterns, poor accessible names that are technically present, or reading order failures.

Screen reader matrix

ToolPlatformNotes
NVDA (free)WindowsMost common — use with Chrome/Firefox
VoiceOvermacOS + iOSBuilt in — Cmd+F5 to toggle
JAWS (paid)WindowsEnterprise / government standard
TalkBackAndroidBuilt in — for mobile testing

7 rules that catch 80% of failures

  • Every interactive element has a keyboard interaction and a visible focus state
  • Every image, icon, and chart has a text alternative
  • No information is conveyed by color alone
  • All text meets contrast minimums — 4.5:1 normal, 3:1 large
  • Layout reflows at 200% zoom with no horizontal scroll
  • Dynamic changes are announced via aria-live without moving focus
  • Focus always returns to the triggering element when an overlay closes

10 / COLUMN NAVIGATION

Column navigation

Multi-column layouts need two independent entry points: landmark jumping (R key in screen readers) and Tab-based header navigation. Both must work independently.

Level AWCAG 2.1.1 · 1.3.1 · 2.4.3

Moving between columns

KeyWhat happensScreen reader says
R (NVDA/JAWS)Jumps to next role="region" landmark — column to column without Tab"FB Post feed, region. 55,900 items."
TabMoves through column headers in DOM order"FB Post, column header, 1 of 3"
Ctrl + →From anywhere in Column 1 — jumps to Column 2 header. Wraps from last to first."FB DM, column header, 2 of 3"
Ctrl + ←Moves to previous column header. Wraps from first to last."FB Post, column header, 1 of 3"
Tab (on header)Enters the column — focus moves to first card inside"Patrick Olsen, Facebook DM, 21 hours ago, 1 of 55,900."
Escape (inside column)Exits column — returns focus to the column header"FB Post, column header, 1 of 3"
Shift + Tab (first card)Returns to column header — same as Escape"FB Post, column header, 1 of 3"

Column header — the anchor point

The column header is the most important focusable element in the layout. It must announce everything a user needs before entering the column.

INFO"FB Post, column 1 of 3, 55,900 items. Press Tab to enter, Enter to open column menu."
  • Column header is an <h2> with tabindex="0" — it is the landmark heading AND a focusable element
  • role="region" aria-labelledby points to this heading — both landmark navigation (R key) and Tab land on the same element
  • Include column position: aria-label="FB Post, column 1 of 3, 55,900 items" — without position, users can't tell which column they're in after reordering
  • After reordering: every column header's aria-label must update to reflect the new position — this is the most commonly missed step in reorder implementations

Navigating within a column

KeyWhat happensScreen reader says

Escape (in detail)Closes detail — focus returns to exact card that opened it"Patrick Olsen, Facebook DM, 21 hours ago. 1 of 55,900."

Layer 1 — column header menu (··· button)
  • The ··· button opens a menu. "Move left" and "Move right" are menu items — minimum required for compliance
  • Announce: "More options for FB Post column, button, collapsed" → "Move right, menu item, 2 of 4"
Layer 2 — keyboard drag (Space + Arrow)

WARNAfter any drop, ALL column header aria-labels must update: "FB DM, column 1 of 3", "FB Post, column 2 of 3", "TW Post, column 3 of 3". The full new order must also be announced via aria-live.

11 / DROPDOWN WITH SEARCH

Dropdown with search

A searchable dropdown is a combobox pattern. Three focus zones — trigger button, search input, listbox — each with distinct keyboard behaviours.

Level AWCAG 2.1.1 · 4.1.2

Zone 1 — trigger button (dropdown closed)

KeyWhat happensScreen reader says
TabLands on trigger. Dropdown stays closed."All types, collapsed, button"
Enter / SpaceOpens dropdown. Focus moves to search input inside."Search, edit text"
↓Opens dropdown AND moves to first option — skips the input"FB Post, 1 of 14, option"
↑Opens dropdown AND moves to last option"My Assignments, 14 of 14, option"
Alt + ↓Opens dropdown without moving focus into it"Expanded"

Zone 2 — search input (dropdown open)

KeyWhat happensScreen reader says
Any characterTypes into search, filters results in real timeAfter debounce: "4 results" or "No results found" via aria-live
↓Moves focus from input into list — lands on first visible option. Does NOT clear the search."FB Post, 1 of 4, option"
↑Moves to last option (wrap). Stays in input if already at top — do NOT wrap to bottom."My Assignments, 14 of 14, option"
EnterSelects highlighted option (if any). If none highlighted: applies filter."FB Post selected. Dropdown closed."
EscapeFirst press: clears search input. Second press: closes dropdown.First: "Search cleared". Second: "All types, collapsed, button"
TabCloses dropdown without selecting. Focus moves to next page element."[Next element]"
Home / EndMoves text cursor in input — browser default. Do NOT override.Cursor position change only

Zone 3 — listbox (focus on an option)

KeyWhat happensScreen reader says
↓Next option. Wraps from last to first."FB DM, 2 of 14, option"
↑ (from first)Returns focus to search input — does NOT wrap to bottom"Search, edit text"
EnterSelects option. Closes dropdown. Focus returns to trigger."FB DM selected. Dropdown closed."
SpaceMulti-select: toggles without closing. Single-select: same as Enter.Multi: "FB DM, checked". Single: "FB DM selected."
EscapeCloses without selecting. Reverts to previous value."All types, collapsed, button"

BackspaceMoves focus back to input, removes last character"Search, edit text"

Multi-select variant

  • Dropdown stays open after each selection — user builds a set
  • Each option needs aria-selected="true/false" — NOT aria-checked. Listbox options always use aria-selected.
  • Trigger must announce count: aria-label="Filter by platform: 3 selected, collapsed, button" — not just a visual badge
  • Include a "Clear all" option keyboard users can reach — first item in list (reachable via Home) or a dedicated button above the list

Critical violations — what must never happen

ERRTab must never navigate through list options. Tab inside dropdown = close and exit. 20 options = 20+ Tab presses to escape. Keyboard trap equivalent.
ERRNever auto-select on Arrow key. Moving through options should only change highlight, not apply selection. Selection happens on Enter or Space only.
ERRNever lose search text when user arrows into list. User types "fb", arrows into results — the input must still show "fb".
ERRDebounce result count announcements. 150–200ms minimum. Without debouncing, typing "facebook" = 8 consecutive announcements that interrupt and disorient.

12 / TABS — FULL SPEC

Tabs — full spec

The entire tab list = one Tab stop. Arrow keys navigate between tabs. Tab enters the panel. This is the most commonly mis-implemented widget.

Level AWCAG 2.1.1 · 2.4.3 · 4.1.2

Complete keyboard map

KeyWhereWhat happens
TabAnywhere on pageLands on the active tab button (whole tablist = 1 Tab stop). Announces: "Overview tab, selected, 1 of 4"
← →On tab listMoves between tabs without entering content. Automatic activation: content updates on arrow. Manual: requires Enter/Space to activate.
Home / EndOn tab listJump to first / last tab. Required by ARIA spec — must be implemented.
TabOn tab listMoves focus INTO the panel — first focusable element inside. If no focusable elements: panel needs tabindex="0".
Shift + TabFirst panel elementReturns focus to the currently active tab button — not first tab, not page top. Requires explicit handling.
Enter / SpaceTab button (manual mode)Activates tab and loads content. Used in manual activation pattern.
EscapeInside panelReturns focus to active tab button. Gives fast exit from deep panel content.

Automatic vs manual activation

Automatic activation
Arrow key moves focus AND activates tab immediately. Content loads as you navigate.
Use when: tab content is already in DOM or loads in <100ms.
Manual activation
Arrow moves focus only. Enter or Space loads content. Prevents multiple API calls while arrowing.
Use when: tabs trigger heavy data loads, API calls, or chart renders.

Screen reader announcements — design these deliberately

MomentScreen reader saysWhat makes it work
Tab lands on list"Overview, tab, selected, 1 of 4"role="tab" + aria-selected="true" + aria-setsize="4" + aria-posinset="1"
Arrow to next tab"Messages, tab, selected, 2 of 4"Focus moves + aria-selected updates
Tab into panel"Messages panel, [first element]"role="tabpanel" + aria-labelledby="tab-messages"
Unread count on tab"Messages, 5 unread, tab, selected, 2 of 4"aria-label="Messages, 5 unread" on the tab button
Disabled tab"Reports, tab, dimmed, 3 of 4"aria-disabled="true" — NOT HTML disabled which removes it from Tab order
Shift+Tab exits panel"Messages, tab, selected, 2 of 4"Explicit keydown handler on first panel element

Required markup

<div role="tablist" aria-label="Conversation sections">
  <button role="tab" aria-selected="true"  tabindex="0"
          aria-controls="panel-1" aria-setsize="4" aria-posinset="1">Overview</button>
  <button role="tab" aria-selected="false" tabindex="-1"
          aria-controls="panel-2" aria-setsize="4" aria-posinset="2">Messages</button>
</div>
<div role="tabpanel" id="panel-1" aria-labelledby="tab-1" tabindex="0">...</div>
<div role="tabpanel" id="panel-2" aria-labelledby="tab-2" tabindex="0" inert>...</div>

Critical violations

ERRNever make each tab button a separate Tab stop. Tablist = 1 Tab stop. Use roving tabindex: active tab = tabindex="0", all others = tabindex="-1".
ERRNever forget aria-selected on inactive tabs. Active: aria-selected="true". All others: aria-selected="false" — explicitly false, not absent.
ERRNever hide inactive panels with display:none. Use the inert attribute instead. display:none breaks focus management when switching tabs.
ERRDon't move focus when tab content loads. The user is still on the tab list — they decide when to enter the panel with Tab.

13 / CREATE MESSAGE FLOW

Create message flow

A concrete end-to-end example of accessible form validation — from first Tab to success announcement. Every step has a defined accessible state.

Level ALevel AAWCAG 3.3.1 · 3.3.3

The complete flow

StepUser actionScreen reader says
1Tab lands on message textarea"Message, required, edit text"
2Type message → Tab → lands on channel dropdown"Channel, required, Select a channel, collapsed, combobox"
3Tab again → skip to Create message button (no channel selected)"Create message, button"
4Enter on button — channel missing — validation fires"Channel, required, invalid, Select a channel. Please select a channel before sending."
5User selects a channel"Facebook Page selected. Channel, valid."
6Tab → button → Enter — all valid"Message sent to Facebook Page."

Step 4 — error fires: what must happen simultaneously

  • Error message appears below the channel field
  • The field gets aria-invalid="true"
  • Focus moves to the channel dropdown — not to the error message, not to the top of the form
INFOFocus moves to the field that needs fixing. The user can act immediately — no extra navigation required.

Step 5 — clear errors eagerly, not lazily

WARNClear the error on change, not on next submit attempt. Waiting until resubmit to clear the error is confusing — the user doesn't know if their fix worked.
// Clear error immediately when user selects a channel
channelInput.addEventListener('change', () => {
  channelInput.setAttribute('aria-invalid', 'false');
  errorEl.textContent = '';
  announce('Facebook Page selected. Channel, valid.');
});

14 / CHECKBOX — SELECT ALL

Checkbox — select all

The select-all checkbox has three states: checked, unchecked, and indeterminate. The indeterminate state must be set via JavaScript — there is no HTML attribute for it.

Level AWCAG 1.3.1 · 4.1.2

Three states

Checked
All items selected. Screen reader: "checked"
Unchecked
No items selected. Screen reader: "not checked"
Indeterminate
Some items selected. Set via JS: checkbox.indeterminate = true. Screen reader announces automatically as "mixed" — no extra ARIA needed.

Required markup

<fieldset>
  <legend>Team members</legend>

  <div>
    <input type="checkbox" id="select-all"
           aria-controls="item1 item2 item3">  ← declares what it controls
    <label for="select-all">Select all</label>
  </div>

  <div>
    <input type="checkbox" id="item1" name="items">
    <label for="item1">Alice</label>
  </div>
  <!-- repeat for item2, item3 ... -->
</fieldset>

JavaScript logic

const selectAll = document.getElementById('select-all');
const items = document.querySelectorAll('input[name="items"]');

selectAll.addEventListener('change', () => {
  items.forEach(item => item.checked = selectAll.checked);
  announce(selectAll.checked ? 'All items selected' : 'All items deselected');
});

function updateSelectAll() {
  const total = items.length;
  const checked = [...items].filter(i => i.checked).length;

  if (checked === 0) {
    selectAll.checked = false;
    selectAll.indeterminate = false;
    announce('No items selected');
  } else if (checked === total) {
    selectAll.checked = true;
    selectAll.indeterminate = false;
    announce('All items selected');
  } else {
    selectAll.checked = false;
    selectAll.indeterminate = true;  ← JS only — no HTML attribute for this
    announce(`${checked} of ${total} items selected`);
  }
}

items.forEach(item => item.addEventListener('change', updateSelectAll));
INFO"2 of 5 selected" via aria-live is far more useful than the checkbox state alone. Always announce the count — don't rely on screen readers interpreting "mixed" as meaningful context.

Key rules

  • Always use a real <input type="checkbox"> — never <div role="checkbox"> unless you have a very good reason. Native elements give keyboard support and screen reader behavior for free.
  • Wrap everything in <fieldset> + <legend> so screen readers announce the group name with each checkbox
  • Use aria-controls on the select-all to declare the relationship — JAWS and NVDA surface this to users
  • The indeterminate state must be set via JS (.indeterminate = true) — there is no HTML attribute for it
  • Announce count changes via aria-live — "2 of 5 selected" is far more useful than just "mixed"
